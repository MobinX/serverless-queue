# Using with AI Agents

`serverless-queue` ships a single dense context file — `llm/llm.txt` — that gives any AI coding agent everything it needs to implement a correct, production-ready integration from scratch. No back-and-forth, no missing context, no hallucinated APIs.

---

## What is llm.txt?

`llm/llm.txt` is a self-contained plain-text reference file. It contains every type definition, every abstract class contract, complete platform examples (Next.js, Cloudflare Workers, Node.js, Bun), full storage implementations (Postgres, Redis, D1, SQLite), and a list of critical invariants and common mistakes — all in a single file you can paste directly into any AI context window.

Instead of asking an agent to read multiple documentation pages, you give it one file. The agent has full context on:

- The exact TypeScript types and interfaces
- What every abstract method must and must not do
- The precise `handle()` invocation flow, step by step
- Complete, working implementations for every major database and platform
- What can go wrong and how to avoid it

This follows the spirit of the [llms.txt](https://llmstxt.org) convention — a standard for providing LLM-readable project documentation as a plain-text file at a predictable path.

<Callout type="tip">
**Paste once, implement correctly.** Most AI agents can fully implement a working `serverless-queue` integration from `llm.txt` alone — storage class, action class, retry strategy, invoker, and platform wiring — in a single prompt.
</Callout>

## GitHub Copilot

### Method 1 — Workspace Instructions

Add a reference to `llm.txt` in `.github/copilot-instructions.md`. Copilot reads this file automatically for every chat and inline suggestion in the workspace.

Create `.github/copilot-instructions.md`:

```markdown
# Copilot Workspace Instructions

This project uses the `serverless-queue` library for background job processing.

The complete reference for implementing serverless-queue integrations (types,
abstract classes, platform examples, storage implementations) is in:

  llm/llm.txt

Read that file before implementing any queue-related code. Pay special attention
to the CRITICAL INVARIANTS section — violating those causes hard-to-debug bugs.
```

With this in place, every Copilot Chat session in VS Code automatically has context about `serverless-queue` without you needing to reference the file manually.

### Method 2 — Chat Context with `#file`

In VS Code Copilot Chat, use `#file:llm/llm.txt` to attach the reference file directly to any prompt:

```
Implement a PostgreSQL MessageStorage for serverless-queue using the patterns in #file:llm/llm.txt
```

```
Wire up a complete Next.js App Router queue endpoint. Use #file:llm/llm.txt for the implementation patterns.
```

```
Write a RetryStrategy that pushes exhausted messages to a DLQ table. See #file:llm/llm.txt for the RetryStrategy contract.
```

### Method 3 — Inline as Context

Open `llm/llm.txt` in VS Code, select all, and use **"Add selection to chat"** from the editor context menu. This injects the full file into the current Copilot Chat session as a context block.

### Example Prompts

## Claude Code

### Using CLAUDE.md

Claude Code reads `CLAUDE.md` at the start of every session. Add a reference to `llm.txt` so Claude always has the full context available:

Create `CLAUDE.md` in your project root:

```markdown
# Claude Project Instructions

## Queue: serverless-queue

This project uses `serverless-queue` for reliable background processing.

The complete implementation reference is at `llm/llm.txt`. This file contains:
- All TypeScript types and interfaces (exact, copy-pasteable)
- Abstract class contracts with full method semantics
- Complete storage implementations (Postgres, Redis, D1, SQLite)
- Platform wiring examples (Next.js, Cloudflare Workers, Node.js, Bun)
- Critical invariants — read these before implementing anything

When implementing queue-related code, read llm/llm.txt first and follow its
patterns exactly, especially:
1. handle() must always be awaited and the HTTP response must always be 200
2. invoke() always receives and sends a QueueMessage[] array (never a single object)
3. writeMessage() must throw on duplicate IDs (use INSERT not UPSERT)
```

### Paste at Session Start

For a one-off session, paste the file content directly:

```bash
cat llm/llm.txt | claude "Using this as context, implement a Cloudflare D1 MessageStorage"
```

## Cursor

### `.cursorrules`

Create `.cursorrules` in your project root to give Cursor's AI access to the reference file for all inline suggestions and Composer sessions:

```
# serverless-queue integration rules

When writing any serverless-queue code (storage, action, retry strategy, invoker,
wiring), always reference the full implementation guide at llm/llm.txt.

Key rules to follow from that file:
- MessageStorage has 7 abstract methods: readMessage, writeMessage, updateMessage,
  deleteMessage, readLastMessageByQueue, readMessagesByQueue, readPendingMessages
- handle() is always awaited and the HTTP handler always returns 200
- invoke() always receives QueueMessage<T>[] (array) — serialize as JSON array
- writeMessage() must throw on duplicate id — never silently overwrite
- DB index on (queue_id, timestamp) is required

For complete type definitions, platform examples, and storage implementations,
read @file llm/llm.txt before generating code.
```

### General Pattern

For any agent that accepts file or URL context:

1. **File path**: Pass `llm/llm.txt` as a context file if the tool supports it.
2. **Raw URL**: Fetch from the URL below and paste as system context.
3. **Paste directly**: Copy the full file content and paste it before your prompt.

The file is plain text with no markdown, so it pastes cleanly into any context window without formatting artifacts.

## Keeping llm.txt in Sync

`llm/llm.txt` lives in the repository and is updated with each release. It reflects the exact types, method signatures, and behavior of the version you have installed.

After upgrading `serverless-queue`, pull the latest `llm.txt` from the repository:

```bash
# If you cloned the repo
git pull && cat llm/llm.txt

# Fetch the latest version directly
curl -o llm/llm.txt https://raw.githubusercontent.com/your-org/serverless-queue/main/llm/llm.txt
```

<Callout type="warning">
Always use the `llm.txt` that matches your installed version. An outdated `llm.txt` may reference method signatures that have changed.
</Callout>
